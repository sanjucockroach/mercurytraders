   108‚Üí   108‚Üí5. General Instructions
   109‚Üí   109‚ÜíThese are non-negotiable project rules. Encode them in your system prompt / AGENTS.md.
   110‚Üí   110‚Üí
   111‚Üí   111‚Üí5.1 Design discipline
   112‚Üí   112‚ÜíOne accent color. Used only for CTAs, focus rings, links. Never decorative. Never a second accent "for variety."
   113‚Üí   113‚ÜíOne radius grammar. Pills (9999px) = consumer/playful. 2‚Äì4px = technical/precision. Never mix.
   114‚Üí   114‚ÜíNegative letter-spacing on display type. Premium brands use -2.5px to -5.5px on large display, ~0 on body. This is a universal signature ‚Äî do not "normalize" it.
   115‚Üí   115‚ÜíOne motion timing. Pick a universal transition duration (e.g. 0.33s Tesla, 0.2s snappy) and use it everywhere. Consistency in motion = consistency in brand.
   116‚Üí   116‚ÜíPick one depth philosophy and commit:
   117‚Üí   117‚ÜíSurface ladder + hairline borders, zero drop shadows (Linear/Framer) ‚Äî preferred for 3D sites (the 3D already provides depth).
   118‚Üí   118‚ÜíOne signature product shadow only (Apple).
   119‚Üí   119‚ÜíStacked subtle shadows (Vercel).
   120‚Üí   120‚ÜíNever use #000000 true black. Use a tinted near-black (#010102, #090909, #1a0d04).
   121‚Üí   121‚ÜíNo "variety for variety's sake." Restraint reads as premium; abundance reads as slop.
   122‚Üí   122‚Üí5.2 Code discipline
   123‚Üí   123‚ÜíNext.js 16 App Router + TypeScript 5. Non-negotiable.
   124‚Üí   124‚Üí"use client" at the top of any file using hooks/browser APIs. Server Components by default.
   125‚Üí   125‚Üíz-ai-web-dev-sdk (and any AI SDK) is backend-only. Never import in client components.
   126‚Üí   126‚ÜíNo backend unless explicitly requested. Default to static. Orders/forms route to WhatsApp / mailto / external endpoints.
   127‚Üí   127‚ÜíImages in /public/images/. Generate premium photography via the image-generation skill; never ship emoji-only placeholders for featured items.
   128‚Üí   128‚ÜíUse existing shadcn/ui components in src/components/ui/ ‚Äî don't rebuild from scratch.
   129‚Üí   129‚ÜíSticky footer mandatory. Root wrapper min-h-screen flex flex-col, footer gets mt-auto. Footer must respect env(safe-area-inset-bottom).
   130‚Üí   130‚Üí5.3 Package manager discipline (Vercel-critical)
   131‚Üí   131‚Üí‚öÝÔ∏è Use npm, not bun, for the deployment lifecycle. Bun is fine for local dev speed, but Vercel's build pipeline and many CI environments are most reliable with npm. The SD Waffles project hit deployment friction from bun-specific lockfile/build differences.

   206‚Üí   206‚Üí7.1 The 4-step adaptation protocol
   207‚Üí   207‚Üítext
   208‚Üí   208‚Üí
   209‚Üí   209‚ÜíSTEP 1 ‚Äî EXTRACT THE BRIEF
   210‚Üí   210‚Üí  - Business name, domain, tagline, vibe, audience, must-have sections.
   211‚Üí   211‚Üí  - Color preferences (if any). If none, propose based on domain archetypes.
   212‚Üí   212‚Üí
   213‚Üí   213‚ÜíSTEP 2 ‚Äî CHOOSE REFERENCES
   214‚Üí   214‚Üí  - Scan ~/playbook-references/awesome-design-md/README.md.
   215‚Üí   215‚Üí  - Pick 1 PRIMARY reference matching the canvas polarity (dark vs light).
   216‚Üí   216‚Üí  - Pick 1 SECONDARY reference matching the depth philosophy (shadows vs surface ladder vs photography-as-depth).
   217‚Üí   217‚Üí  - Copy both DESIGN.md files into the project as DESIGN-ref-1.md + DESIGN-ref-2.md.
   218‚Üí   218‚Üí
   219‚Üí   219‚ÜíSTEP 3 ‚Äî SYNTHESIZE PROJECT DESIGN.md
   220‚Üí   220‚Üí  - Start from the template in ¬ß6.1.
   221‚Üí   221‚Üí  - Inherit the canvas color, accent discipline, and radius grammar from PRIMARY.
   222‚Üí   222‚Üí  - Inherit the depth philosophy and motion timing from SECONDARY.
   223‚Üí   223‚Üí  - Override palette + typography with the user's domain colors/fonts.
   224‚Üí   224‚Üí  - Write the "Do's and Don'ts" as hard constraints (these become lint rules for the agent).
   225‚Üí   225‚Üí
   226‚Üí   226‚ÜíSTEP 4 ‚Äî VALIDATE BEFORE CODING
   227‚Üí   227‚Üí  - Read DESIGN.md aloud. Does it commit to ONE accent? ONE radius grammar? ONE depth philosophy?
   228‚Üí   228‚Üí  - If any answer is "both" or "mixed," revise. Restraint = premium.

   278‚Üí   278‚Üí8.2 The 7 rules of a 3D hero
   279‚Üí   279‚Üí3D frames the text, never fights it. Objects on the sides/at edges; center clear. If text overlaps 3D, add a radial gradient veil behind the text.
   280‚Üí   280‚ÜíOne focal point. The eye lands on the tagline first, then the 3D, then the CTA. Don't compete.
   281‚Üí   281‚ÜíNegative tracking on the tagline. -2.5px to -5.5px depending on size. This is the single biggest "premium" tell.
   282‚Üí   282‚ÜíGradient accent on 1-2 words only (e.g. "waffles" in gold, "shakes" in gold, "&" in pink). Never gradient the whole line.
   283‚Üí   283‚ÜíGlassmorphism for overlays ‚Äî backdrop-blur + semi-transparent bg + 1px hairline border. Use for eyebrow pills, stat cards, navbar on scroll.
   284‚Üí   284‚Üíprefers-reduced-motion fallback ‚Äî static poster image instead of 3D animation. Mandatory for a11y.
   285‚Üí   285‚ÜíLCP element is the tagline <h1>. Font display: swap, preload the display font. Scene initializes after first paint.

   490‚Üí   490‚Üí  REFINE: fix every CRITIQUE point
   491‚Üí   491‚Üí  EXIT when: renders + matches DESIGN.md + mobile-OK + a11y-OK

   517‚Üí   517‚Üí12. Looping Prompts (Loop Engineering)
   518‚Üí   518‚ÜíLoop engineering is the practice of building an iterative workflow around an LLM instead of relying on a single prompt. The agent repeatedly reasons ‚Üí acts ‚Üí observes ‚Üí reflects ‚Üí refines until a verifiable success criterion is met, with the human defining the rules of the loop and the agent handling execution.
   519‚Üí   519‚Üí
   520‚Üí   520‚ÜíGrounded in: ReAct (Yao et al. 2022), Reflexion (Shinn et al. 2023), Andrew Ng's Reflection pattern, Anthropic's Evaluator-Optimizer, and Steve Kinney's "Anatomy of an Agent Loop."
   521‚Üí   521‚Üí
   522‚Üí   522‚Üí12.1 The canonical loop structure
   523‚Üí   523‚Üítext
   524‚Üí   524‚Üí
   525‚Üí   525‚Üí1. ASSEMBLE CONTEXT ‚Äî system prompt, goal, state, prior outputs, memory
   526‚Üí   526‚Üí2. PLAN / REASON   ‚Äî decompose goal, decide next action (Thought)
   527‚Üí   527‚Üí3. ACT             ‚Äî emit tool call / code edit / candidate output (Action)
   528‚Üí   528‚Üí4. OBSERVE         ‚Äî harness executes, feeds result back (Observation)
   529‚Üí   529‚Üí5. EVALUATE        ‚Äî score against verifiable success criterion
   530‚Üí   530‚Üí6. REFLECT / REFINE‚Äî self-critique, store lesson, revise
   531‚Üí   531‚Üí7. CHECK TERMINATION ‚Äî if success predicate true OR guard trips, exit; else repeat
   532‚Üí   532‚ÜíThe whole thing, in code:
   533‚Üí   533‚Üí
   534‚Üí   534‚Üíjs
   535‚Üí   535‚Üí
   536‚Üí   536‚Üíwhile (!done) {
   537‚Üí   537‚Üí  const response = await callLLM(messages);
   538‚Üí   538‚Üí  if (response.toolCalls.length > 0) {
   539‚Üí   539‚Üí    const results = await executeTools(response.toolCalls);
   540‚Üí   540‚Üí    messages.push(...results);      // tool call = "I'm not done yet"
   541‚Üí   541‚Üí  } else {
   542‚Üí   542‚Üí    done = true;                     // text-only response = termination signal
   543‚Üí   543‚Üí    return response;
   544‚Üí   544‚Üí  }
   545‚Üí   545‚Üí}
   546‚Üí   546‚Üí12.2 Termination guards (defense in depth ‚Äî ALWAYS layer these)
   547‚Üí   547‚ÜíMax iterations (15‚Äì25 typical)
   548‚Üí   548‚ÜíWall-clock timeout (~300s)
   549‚Üí   549‚ÜíToken/cost budget (~$2/run)
   550‚Üí   550‚ÜíNo-progress detection ‚Äî hash each iteration's (tool, result_preview); break on 3 identical fingerprints
   551‚Üí   551‚ÜíOn hitting the cap ‚Äî don't stop silently; append "You've reached the maximum steps; give your best answer now" and call the LLM one final time without tools
   552‚Üí   552‚Üí12.3 Five loop-prompt templates (copy-pasteable)
   553‚Üí   553‚ÜíTemplate A ‚Äî Build-Component Loop (use for every non-trivial component)
   554‚Üí   554‚Üítext
   555‚Üí   555‚Üí
   556‚Üí   556‚ÜíYou are in a build-component loop. Repeat until done.
   557‚Üí   557‚Üí
   558‚Üí   558‚ÜíOBJECTIVE: Build <component> matching DESIGN.md.
   559‚Üí   559‚ÜíSTOP CONDITION: component renders in Agent Browser with 0 console
   560‚Üí   560‚Üíerrors, matches DESIGN.md accent/radius/motion rules, mobile-legible,
   561‚Üí   561‚Üía11y-OK; OR max_iterations=10; OR 3 identical (action, result) fingerprints.
   562‚Üí   562‚Üí
   563‚Üí   563‚ÜíEach iteration, output:
   564‚Üí   564‚Üí  PLAN:      what you're adding/changing and why.
   565‚Üí   565‚Üí  ACT:       one concrete edit (surgical, not full-file rewrite).
   566‚Üí   566‚Üí  OBSERVE:   paste the relevant dev-server output / screenshot finding.
   567‚Üí   567‚Üí  CRITIQUE:  check against DESIGN.md ‚Äî accent? radius? motion? mobile? a11y?
   568‚Üí   568‚Üí  REFLECT:   what to try next, or "DONE" if all criteria pass.
   569‚Üí   569‚Üí
   570‚Üí   570‚ÜíRules:
   571‚Üí   571‚Üí- Re-read DESIGN.md every iteration; do not drift.
   572‚Üí   572‚Üí- Keep only the latest error/screenshot in context.
   573‚Üí   573‚Üí- If a command errors, treat it as feedback ‚Äî diagnose and retry differently.
   574‚Üí   574‚Üí- Never declare DONE without an Agent Browser screenshot proving it renders.
   575‚Üí   575‚ÜíTemplate B ‚Äî TDD / Bug-Fix Loop (use when something is broken)
   576‚Üí   576‚Üítext
   577‚Üí   577‚Üí
   578‚Üí   578‚ÜíYou are an autonomous coding agent in a TDD loop. Repeat until done.
   579‚Üí   579‚Üí
   580‚Üí   580‚ÜíOBJECTIVE: Make the failing test / build / lint pass.
   581‚Üí   581‚ÜíSTOP CONDITION: `npm run build` exits 0 AND `npx tsc --noEmit` exits 0
   582‚Üí   582‚ÜíAND `npm run lint` exits 0; OR max_iterations=15; OR 3 identical
   583‚Üí   583‚Üí(tool, result) fingerprints (you are stuck).
   584‚Üí   584‚Üí
   585‚Üí   585‚ÜíEach iteration, output:
   586‚Üí   586‚Üí  PLAN:      which error you are targeting and why it fails.
   587‚Üí   587‚Üí  ACT:       one surgical edit (line-replace, not full-file rewrite).
   588‚Üí   588‚Üí  OBSERVE:   paste the relevant slice of the new build/lint output.
   589‚Üí   589‚Üí  REFLECT:   what changed, what to try next, or "DONE" if all green.
   590‚Üí   590‚Üí
   591‚Üí   591‚ÜíRules:
   592‚Üí   592‚Üí- Keep only the latest error messages in context; summarize prior iterations.
   593‚Üí   593‚Üí- Re-read the OBJECTIVE every iteration; do not drift into unrelated refactors.
   594‚Üí   594‚Üí- If a command errors (missing dep, bad env), treat the error as feedback ‚Äî
   595‚Üí   595‚Üí  diagnose and retry a different approach; do not stop.
   596‚Üí   596‚Üí- On the final iteration if still failing, stop and summarize your best progress.
   597‚Üí   597‚ÜíTemplate C ‚Äî Verification / QA Loop (evaluator-optimizer, two-role)
   598‚Üí   598‚Üítext
   599‚Üí   599‚Üí
   600‚Üí   600‚Üí# GENERATOR
   601‚Üí   601‚ÜíProduce <artifact: section/component/page>. Output it inside
   602‚Üí   602‚Üí<artifact>...</artifact> tags, followed by a <self_check> block listing
   603‚Üí   603‚Üíwhich acceptance criteria are met and which are not (reason first,
   604‚Üí   604‚Üíthen output).
   605‚Üí   605‚Üí
   606‚Üí   606‚Üí# EVALUATOR (separate LLM call, given artifact + criteria)
   607‚Üí   607‚ÜíCriteria:
   608‚Üí   608‚Üí  1. scene loads in <2s on mobile (verifiable via Agent Browser)
   609‚Üí   609‚Üí  2. no console errors (verifiable)
   610‚Üí   610‚Üí  3. one accent color only (verifiable via screenshot)
   611‚Üí   611‚Üí  4. negative tracking on display type (verifiable via screenshot)
   612‚Üí   612‚Üí  5. mobile-legible at 390px (verifiable via screenshot)
   613‚Üí   613‚Üí  6. prefers-reduced-motion respected (verifiable via code review)
   614‚Üí   614‚ÜíVerdict per criterion: PASS | FAIL + one-sentence reason.
   615‚Üí   615‚ÜíOverall: ACCEPT (all pass) or REJECT.
   616‚Üí   616‚ÜíIf REJECT, emit a structured <feedback> block the generator must address.
   617‚Üí   617‚Üí
   618‚Üí   618‚Üí# LOOP CONTROL (harness)
   619‚Üí   619‚Üíwhile not all(criteria == PASS) and iteration < max_iters:
   620‚Üí   620‚Üí    artifact = generator(task, feedback)
   621‚Üí   621‚Üí    verdict, feedback = evaluator(artifact, criteria)
   622‚Üí   622‚ÜíTemplate D ‚Äî Design-Refinement Loop (self-critique with exit gate)
   623‚Üí   623‚Üítext
   624‚Üí   624‚Üí
   625‚Üí   625‚ÜíYou are iterating on a design spec / DESIGN.md for a 3D website feature.
   626‚Üí   626‚Üí
   627‚Üí   627‚ÜíRound format (repeat up to 8 times):
   628‚Üí   628‚Üí  DRAFT:    your current best spec.
   629‚Üí   629‚Üí  CRITIQUE: act as a strict senior reviewer. List concrete defects against
   630‚Üí   630‚Üí            the rubric: performance, accessibility, build feasibility,
   631‚Üí   631‚Üí            scope creep, edge cases. Be specific; do not rubber-stamp.
   632‚Üí   632‚Üí  SCORE:    1‚Äì5 per rubric dimension.
   633‚Üí   633‚Üí  REVISE:   produce the next DRAFT addressing every CRITIQUE point.
   634‚Üí   634‚Üí
   635‚Üí   635‚ÜíEXIT when: every rubric dimension scores >= 4 AND the CRITIQUE block
   636‚Üí   636‚Üícontains no "blocking" items, OR round 8 is reached.
   637‚Üí   637‚ÜíOn exit, output the final spec prefixed with "FINAL:".
   638‚Üí   638‚ÜíTemplate E ‚Äî ReAct (canonical, for research / multi-step exploration)
   639‚Üí   639‚Üítext
   640‚Üí   640‚Üí
   641‚Üí   641‚ÜíQuestion: <goal>
   642‚Üí   642‚ÜíThought 1: <decompose, decide first action>
   643‚Üí   643‚ÜíAction 1: <tool call, e.g. Search[...] or page_reader[...]>
   644‚Üí   644‚ÜíObservation 1: <result>
   645‚Üí   645‚ÜíThought 2: <reason about observation>
   646‚Üí   646‚ÜíAction 2: <next action>
   647‚Üí   647‚ÜíObservation 2: <result>
   648‚Üí   648‚Üí...
   649‚Üí   649‚ÜíThought N: <synthesis>
   650‚Üí   650‚ÜíAction N: Finish[<final answer>]
   651‚Üí   651‚ÜíNote: Finish[...] is the explicit termination signal. Without it, the model keeps going.
   652‚Üí   652‚Üí
   653‚Üí   653‚Üí12.4 Best practices
   654‚Üí   654‚ÜíDefine explicit, verifiable exit criteria. "Tests pass," "lint clean," "Agent Browser shows 0 errors" ‚Äî not "the model stopped."
   655‚Üí   655‚ÜíSeparate generator from evaluator when possible; distinct roles reduce sycophancy.
   656‚Üí   656‚ÜíEmit reasoning/feedback blocks before final output to trigger chain-of-thought.
   657‚Üí   657‚ÜíManage context aggressively ‚Äî keep only latest errors, paginate large reads, summarize prior iterations, re-inject the objective every cycle.
   658‚Üí   658‚ÜíTreat errors as feedback, not termination ‚Äî feed the error back so the model can diagnose and retry.
   659‚Üí   659‚ÜíLog structured traces per run: iteration count, tokens/turn, tool calls, final result.
   660‚Üí   660‚ÜíMake side-effecting tools idempotent so retries don't double-fire.
   661‚Üí   661‚Üí12.5 Pitfalls to avoid
   662‚Üí   662‚ÜíInfinite loops ‚Äî the #1 production failure. Always layer termination guards.
   663‚Üí   663‚ÜíContext-window overflow ‚Äî tool results are the biggest token consumer. Compact aggressively.
   664‚Üí   664‚ÜíTreating "model stopped" as "goal met" ‚Äî a terminal message may be a clarifying question. Always run an explicit goal-completion check.
   665‚Üí   665‚ÜíDrift ‚Äî agents look productive early, then gradually lose context and make disconnected decisions while remaining confident. Fix with tight scope + hard stop conditions.
   666‚Üí   666‚ÜíError compounding ‚Äî a mistake at step 4 propagates silently into a confidently-wrong result by step 18. Use shorter chains with verification checkpoints.
   667‚Üí   667‚ÜíTool confusion ‚Äî too many overlapping tools cause the agent to pick the wrong one. Fewer tools, explicit namespacing.
   668‚Üí   668‚ÜíMissing idempotency ‚Äî retrying after a timeout double-fires side effects. Mandatory for any side-effecting tool.
