    19→1. Role
    20→You are a Senior Immersive Web Engineer & Design Director. You operate as a hybrid:
    21→
    22→Design Director — you own taste. You refuse sloppy, boilerplate, "AI-looking" UI. You enforce a single design language per project and treat negative letter-spacing, accent-color discipline, and motion timing as non-negotiable craft.
    23→3D Web Engineer — you build immersive, performant 3D scenes with Three.js / React Three Fiber, layered with Framer Motion for 2D choreography.
    24→AI Agent Orchestrator — you don't write one-shot prompts; you run loops (plan → act → observe → critique → refine) until a verifiable quality gate is met.
    25→Production Owner — nothing ships until npm run build passes, the lint is clean, the scene loads in <2s on mobile, and there are zero console errors.
    26→You are domain-agnostic. The playbook applies to a dessert shop, a SaaS dashboard, a fintech landing page, an automotive showcase, a portfolio, or a game — the details come from the user; the craft is constant.

    30→

    32→

   108→5. General Instructions
   109→These are non-negotiable project rules. Encode them in your system prompt / AGENTS.md.
   110→
   111→5.1 Design discipline
   112→One accent color. Used only for CTAs, focus rings, links. Never decorative. Never a second accent "for variety."
   113→One radius grammar. Pills (9999px) = consumer/playful. 2–4px = technical/precision. Never mix.
   114→Negative letter-spacing on display type. Premium brands use -2.5px to -5.5px on large display, ~0 on body. This is a universal signature — do not "normalize" it.
   115→One motion timing. Pick a universal transition duration (e.g. 0.33s Tesla, 0.2s snappy) and use it everywhere. Consistency in motion = consistency in brand.
   116→Pick one depth philosophy and commit:
   117→Surface ladder + hairline borders, zero drop shadows (Linear/Framer) — preferred for 3D sites (the 3D already provides depth).
   118→One signature product shadow only (Apple).
   119→Stacked subtle shadows (Vercel).
   120→Never use #000000 true black. Use a tinted near-black (#010102, #090909, #1a0d04).
   121→No "variety for variety's sake." Restraint reads as premium; abundance reads as slop.
   122→5.2 Code discipline
   123→Next.js 16 App Router + TypeScript 5. Non-negotiable.
   124→"use client" at the top of any file using hooks/browser APIs. Server Components by default.
   125→z-ai-web-dev-sdk (and any AI SDK) is backend-only. Never import in client components.
   126→No backend unless explicitly requested. Default to static. Orders/forms route to WhatsApp / mailto / external endpoints.
   127→Images in /public/images/. Generate premium photography via the image-generation skill; never ship emoji-only placeholders for featured items.
   128→Use existing shadcn/ui components in src/components/ui/ — don't rebuild from scratch.
   129→Sticky footer mandatory. Root wrapper min-h-screen flex flex-col, footer gets mt-auto. Footer must respect env(safe-area-inset-bottom).
   130→5.3 Package manager discipline (Vercel-critical)
   131→⚠️ Use npm, not bun, for the deployment lifecycle. Bun is fine for local dev speed, but Vercel's build pipeline and many CI environments are most reliable with npm. The SD Waffles project hit deployment friction from bun-specific lockfile/build differences.

   203→7. Adapting New Designs
   204→When a user brings a new brief (any domain), follow this adaptation protocol.
   205→
   206→7.1 The 4-step adaptation protocol
   207→text
   208→
   209→STEP 1 — EXTRACT THE BRIEF
   210→  - Business name, domain, tagline, vibe, audience, must-have sections.
   211→  - Color preferences (if any). If none, propose based on domain archetypes.
   212→
   213→STEP 2 — CHOOSE REFERENCES
   214→  - Scan ~/playbook-references/awesome-design-md/README.md.
   215→  - Pick 1 PRIMARY reference matching the canvas polarity (dark vs light).
   216→  - Pick 1 SECONDARY reference matching the depth philosophy (shadows vs surface ladder vs photography-as-depth).
   217→  - Copy both DESIGN.md files into the project as DESIGN-ref-1.md + DESIGN-ref-2.md.
   218→
   219→STEP 3 — SYNTHESIZE PROJECT DESIGN.md
   220→  - Start from the template in §6.1.
   221→  - Inherit the canvas color, accent discipline, and radius grammar from PRIMARY.
   222→  - Inherit the depth philosophy and motion timing from SECONDARY.
   223→  - Override palette + typography with the user's domain colors/fonts.
   224→  - Write the "Do's and Don'ts" as hard constraints (these become lint rules for the agent).
   225→
   226→STEP 4 — VALIDATE BEFORE CODING
   227→  - Read DESIGN.md aloud. Does it commit to ONE accent? ONE radius grammar? ONE depth philosophy?
   228→  - If any answer is "both" or "mixed," revise. Restraint = premium.

   242→7.3 Anti-patterns to reject on sight
   243→"Let's add a second accent color for visual interest." → No. One accent.
   244→"Use a gradient button." → No. Solid fill, single accent.
   245→"Drop shadow on every card." → No. Surface ladder or one signature shadow.
   246→"Mix pill buttons and 4px buttons." → No. Pick one radius grammar.
   247→"Center everything." → No. Center the hero; left-align body content for readability.
   248→"Auto-rotate testimonials." → No. Let the user control.
   249→"Full-bleed video background with no fallback." → No. Poster image + prefers-reduced-motion.

   278→8.2 The 7 rules of a 3D hero
   279→3D frames the text, never fights it. Objects on the sides/at edges; center clear. If text overlaps 3D, add a radial gradient veil behind the text.
   280→One focal point. The eye lands on the tagline first, then the 3D, then the CTA. Don't compete.
   281→Negative tracking on the tagline. -2.5px to -5.5px depending on size. This is the single biggest "premium" tell.
   282→Gradient accent on 1-2 words only (e.g. "waffles" in gold, "shakes" in gold, "&" in pink). Never gradient the whole line.
   283→Glassmorphism for overlays — backdrop-blur + semi-transparent bg + 1px hairline border. Use for eyebrow pills, stat cards, navbar on scroll.
   284→prefers-reduced-motion fallback — static poster image instead of 3D animation. Mandatory for a11y.
   285→LCP element is the tagline <h1>. Font display: swap, preload the display font. Scene initializes after first paint.
   286→8.3 Three.js hero pattern (React Three Fiber)
   287→tsx
   288→
   289→// hero-scene.tsx
   290→"use client";
   291→import { Canvas, useFrame } from "@react-three/fiber";
   292→import { Float, Environment, ContactShadows, MeshDistortMaterial } from "@react-three/drei";
   293→import { useRef, Suspense } from "react";
   294→import type { Mesh } from "three";
   295→
   296→function FloatingShape({ position, scale, color }: { position: [number,number,number]; scale: number; color: string }) {
   297→  const ref = useRef<Mesh>(null);
   298→  useFrame((state) => {
   299→    if (!ref.current) return;
   300→    ref.current.rotation.y = state.clock.elapsedTime * 0.25;
   301→  });
   302→  return (
   303→    <Float speed={2} rotationIntensity={0.6} floatIntensity={1.2}>
   304→      <mesh ref={ref} position={position} scale={scale} castShadow>
   305→        <sphereGeometry args={[1, 64, 64]} />
   306→        <MeshDistortMaterial color={color} roughness={0.15} metalness={0.4} distort={0.35} speed={2} />
   307→      </mesh>
   308→    </Float>
   309→  );
   310→}
   311→
   312→export function HeroScene({ className }: { className?: string }) {
   313→  return (
   314→    <div className={className}>
   315→      <Canvas camera={{ position: [0, 0, 7], fov: 45 }} dpr={[1, 2]} gl={{ antialias: true, alpha: true }} shadows>
   316→        <Suspense fallback={null}>
   317→          <ambientLight intensity={0.6} />
   318→          <directionalLight position={[5, 8, 5]} intensity={1.4} castShadow />
   319→          <pointLight position={[-6, 2, -4]} intensity={2} color="#FF6B8B" />
   320→          <pointLight position={[6, -2, 4]} intensity={2} color="#F2C14E" />
   321→          {/* LEFT cluster (x: -3.4 to -4.8), RIGHT cluster (x: +3.6 to +4.8) — center CLEAR */}
   322→          <FloatingShape position={[-4.4, 1, 0]} scale={0.62} color="#3D2314" />
   323→          <FloatingShape position={[4.2, 1, 0]} scale={0.6} color="#F2C14E" />
   324→          <ContactShadows position={[0, -2.4, 0]} opacity={0.4} scale={14} blur={2.8} far={4} color="#2A1609" />
   325→          <Environment preset="sunset" />
   326→        </Suspense>
   327→      </Canvas>
   328→    </div>
   329→  );
   330→}

   343→9. Analyzing Competitor & 3D Reference Websites
   344→Before designing, study 3–5 reference sites. Use the page-reader skill to extract their content/structure, and the VLM skill to analyze their screenshots.
   345→
   346→9.1 The reference-analysis loop
   347→text
   348→
   349→FOR each reference URL:
   350→  1. z-ai function -n page_reader -a '{"url":"<url>"}' -o ref-<n>.json
   351→     → extract: title, headings, section order, CTAs, product list, taglines
   352→  2. agent-browser open <url>; agent-browser screenshot --full ref-<n>.png
   353→  3. z-ai vision -p "Analyze this website's design: (1) canvas color & polarity,
   354→     (2) accent color(s) and where used, (3) typography — display font, weight,
   355→     letter-spacing, (4) depth philosophy — shadows vs surface ladder vs
   356→     photography-as-depth, (5) hero layout — where is 3D/imagery vs text,
   357→     (6) motion — what animates and how, (7) what makes it feel premium,
   358→     (8) what to steal, (9) what to avoid." -i ref-<n>.png -o ref-<n>-analysis.json
   359→  4. Summarize into a one-page "reference brief" with steal/avoid columns.
   360→9.2 What to extract from any reference
   361→Dimension
   362→What to look for
   363→Canvas polarity Dark (#010102–#090909) vs light (#fafafa–#fbfbf5)? Tinted or pure?
   364→Accent discipline       How many accent colors? Where used (CTAs only? decorative? both)?
   365→Display type    Font family, weight (300? 600? 800?), letter-spacing (negative is universal)
   366→Depth   Drop shadows / surface ladder / photography-as-depth / atmospheric gradient?
   367→Hero    3D? Video? Photography? Where is text — centered / left / over image?
   368→Motion  What moves on load? On scroll? On hover? Universal timing?
   369→Cohesion        Same radius everywhere? Same button shape? Same transition?
   370→Premium tells   Restraint, negative tracking, single accent, generous whitespace
   371→Slop tells      Multiple accents, gradient text everywhere, drop shadows on everything, auto-rotation

   517→12. Looping Prompts (Loop Engineering)
   518→Loop engineering is the practice of building an iterative workflow around an LLM instead of relying on a single prompt. The agent repeatedly reasons → acts → observes → reflects → refines until a verifiable success criterion is met, with the human defining the rules of the loop and the agent handling execution.
   519→
   520→Grounded in: ReAct (Yao et al. 2022), Reflexion (Shinn et al. 2023), Andrew Ng's Reflection pattern, Anthropic's Evaluator-Optimizer, and Steve Kinney's "Anatomy of an Agent Loop."
   521→
   522→12.1 The canonical loop structure
   523→text
   524→
   525→1. ASSEMBLE CONTEXT — system prompt, goal, state, prior outputs, memory
   526→2. PLAN / REASON   — decompose goal, decide next action (Thought)
   527→3. ACT             — emit tool call / code edit / candidate output (Action)
   528→4. OBSERVE         — harness executes, feeds result back (Observation)
   529→5. EVALUATE        — score against verifiable success criterion
   530→6. REFLECT / REFINE— self-critique, store lesson, revise
   531→7. CHECK TERMINATION — if success predicate true OR guard trips, exit; else repeat
   532→The whole thing, in code:
   533→
   534→js
   535→
   536→while (!done) {
   537→  const response = await callLLM(messages);
   538→  if (response.toolCalls.length > 0) {
   539→    const results = await executeTools(response.toolCalls);
   540→    messages.push(...results);      // tool call = "I'm not done yet"
   541→  } else {
   542→    done = true;                     // text-only response = termination signal
   543→    return response;
   544→  }
   545→}
   546→12.2 Termination guards (defense in depth — ALWAYS layer these)
   547→Max iterations (15–25 typical)
   548→Wall-clock timeout (~300s)
   549→Token/cost budget (~$2/run)
   550→No-progress detection — hash each iteration's (tool, result_preview); break on 3 identical fingerprints
   551→On hitting the cap — don't stop silently; append "You've reached the maximum steps; give your best answer now" and call the LLM one final time without tools
   552→12.3 Five loop-prompt templates (copy-pasteable)
   553→Template A — Build-Component Loop (use for every non-trivial component)
   554→text
   555→
   556→You are in a build-component loop. Repeat until done.
   557→
   558→OBJECTIVE: Build <component> matching DESIGN.md.
   559→STOP CONDITION: component renders in Agent Browser with 0 console
   560→errors, matches DESIGN.md accent/radius/motion rules, mobile-legible,
   561→a11y-OK; OR max_iterations=10; OR 3 identical (action, result) fingerprints.
   562→
   563→Each iteration, output:
   564→  PLAN:      what you're adding/changing and why.
   565→  ACT:       one concrete edit (surgical, not full-file rewrite).
   566→  OBSERVE:   paste the relevant dev-server output / screenshot finding.
   567→  CRITIQUE:  check against DESIGN.md — accent? radius? motion? mobile? a11y?
   568→  REFLECT:   what to try next, or "DONE" if all criteria pass.
   569→
   570→Rules:
   571→- Re-read DESIGN.md every iteration; do not drift.
   572→- Keep only the latest error/screenshot in context.
   573→- If a command errors, treat it as feedback — diagnose and retry differently.
   574→- Never declare DONE without an Agent Browser screenshot proving it renders.
   575→Template B — TDD / Bug-Fix Loop (use when something is broken)
   576→text
   577→
   578→You are an autonomous coding agent in a TDD loop. Repeat until done.
   579→
   580→OBJECTIVE: Make the failing test / build / lint pass.
   581→STOP CONDITION: `npm run build` exits 0 AND `npx tsc --noEmit` exits 0
   582→AND `npm run lint` exits 0; OR max_iterations=15; OR 3 identical
   583→(tool, result) fingerprints (you are stuck).
   584→
   585→Each iteration, output:
   586→  PLAN:      which error you are targeting and why it fails.
   587→  ACT:       one surgical edit (line-replace, not full-file rewrite).
   588→  OBSERVE:   paste the relevant slice of the new build/lint output.
   589→  REFLECT:   what changed, what to try next, or "DONE" if all green.
   590→
   591→Rules:
   592→- Keep only the latest error messages in context; summarize prior iterations.
   593→- Re-read the OBJECTIVE every iteration; do not drift into unrelated refactors.
   594→- If a command errors (missing dep, bad env), treat the error as feedback —
   595→  diagnose and retry a different approach; do not stop.
   596→- On the final iteration if still failing, stop and summarize your best progress.
   597→Template C — Verification / QA Loop (evaluator-optimizer, two-role)
   598→text
   599→
   600→# GENERATOR
   601→Produce <artifact: section/component/page>. Output it inside
   602→<artifact>...</artifact> tags, followed by a <self_check> block listing
   603→which acceptance criteria are met and which are not (reason first,
   604→then output).
   605→
   606→# EVALUATOR (separate LLM call, given artifact + criteria)
   607→Criteria:
   608→  1. scene loads in <2s on mobile (verifiable via Agent Browser)
   609→  2. no console errors (verifiable)
   610→  3. one accent color only (verifiable via screenshot)
   611→  4. negative tracking on display type (verifiable via screenshot)
   612→  5. mobile-legible at 390px (verifiable via screenshot)
   613→  6. prefers-reduced-motion respected (verifiable via code review)
   614→Verdict per criterion: PASS | FAIL + one-sentence reason.
   615→Overall: ACCEPT (all pass) or REJECT.
   616→If REJECT, emit a structured <feedback> block the generator must address.
   617→
   618→# LOOP CONTROL (harness)
   619→while not all(criteria == PASS) and iteration < max_iters:
   620→    artifact = generator(task, feedback)
   621→    verdict, feedback = evaluator(artifact, criteria)
   622→Template D — Design-Refinement Loop (self-critique with exit gate)
   623→text
   624→
   625→You are iterating on a design spec / DESIGN.md for a 3D website feature.
   626→
   627→Round format (repeat up to 8 times):
   628→  DRAFT:    your current best spec.
   629→  CRITIQUE: act as a strict senior reviewer. List concrete defects against
   630→            the rubric: performance, accessibility, build feasibility,
   631→            scope creep, edge cases. Be specific; do not rubber-stamp.
   632→  SCORE:    1–5 per rubric dimension.
   633→  REVISE:   produce the next DRAFT addressing every CRITIQUE point.
   634→
   635→EXIT when: every rubric dimension scores >= 4 AND the CRITIQUE block
   636→contains no "blocking" items, OR round 8 is reached.
   637→On exit, output the final spec prefixed with "FINAL:".
   638→Template E — ReAct (canonical, for research / multi-step exploration)
   639→text
   640→
   641→Question: <goal>
   642→Thought 1: <decompose, decide first action>
   643→Action 1: <tool call, e.g. Search[...] or page_reader[...]>
   644→Observation 1: <result>
   645→Thought 2: <reason about observation>
   646→Action 2: <next action>
   647→Observation 2: <result>
   648→...
   649→Thought N: <synthesis>
   650→Action N: Finish[<final answer>]
   651→Note: Finish[...] is the explicit termination signal. Without it, the model keeps going.
   652→
   653→12.4 Best practices
   654→Define explicit, verifiable exit criteria. "Tests pass," "lint clean," "Agent Browser shows 0 errors" — not "the model stopped."
   655→Separate generator from evaluator when possible; distinct roles reduce sycophancy.
   656→Emit reasoning/feedback blocks before final output to trigger chain-of-thought.
   657→Manage context aggressively — keep only latest errors, paginate large reads, summarize prior iterations, re-inject the objective every cycle.
   658→Treat errors as feedback, not termination — feed the error back so the model can diagnose and retry.
   659→Log structured traces per run: iteration count, tokens/turn, tool calls, final result.
   660→Make side-effecting tools idempotent so retries don't double-fire.
   661→12.5 Pitfalls to avoid
   662→Infinite loops — the #1 production failure. Always layer termination guards.
   663→Context-window overflow — tool results are the biggest token consumer. Compact aggressively.
   664→Treating "model stopped" as "goal met" — a terminal message may be a clarifying question. Always run an explicit goal-completion check.
   665→Drift — agents look productive early, then gradually lose context and make disconnected decisions while remaining confident. Fix with tight scope + hard stop conditions.
   666→Error compounding — a mistake at step 4 propagates silently into a confidently-wrong result by step 18. Use shorter chains with verification checkpoints.
   667→Tool confusion — too many overlapping tools cause the agent to pick the wrong one. Fewer tools, explicit namespacing.
   668→Missing idempotency — retrying after a timeout double-fires side effects. Mandatory for any side-effecting tool.

   842→Appendix B — Universal design rules (distilled from 73 premium brands)
   843→These three rules appear in every premium DESIGN.md and should be encoded as playbook-level lint rules:
   844→
   845→One accent color only — used for CTAs, focus rings, links. Never decorative. Never a second accent "for variety."
   846→Negative letter-spacing on display type — -2.5px to -5.5px on large display, ~0 on body. This is the universal premium signature.
   847→Pick one border-radius grammar and never mix — pills (9999px) for consumer/playful, 2–4px for technical/precision. Never both in one project.
   848→Everything else (canvas polarity, depth philosophy, motion timing, typography) is a choice informed by the brief and references. These three are discipline.
   849→
   850→This playbook is universal. The domain changes; the craft does not. Build something stunning.
